# Checkpoint 2

<h2>Background</h2>
<p>Your Checkpoint 1 implementation successfully demonstrated course data management and basic querying. The UBC Decision Support team is impressed with your progress and has expanded the scope: they now need to analyze <strong>facilities data</strong> (buildings and room information) in addition to course offerings.</p>
<p>The team has provided a new dataset format containing building locations, room details, and seating information extracted from the UBC campus facilities system. This expansion requires not only new data processing logic but also more sophisticated query capabilities to support analysis across multiple data types.</p>
<p>Links:</p>
<ul>
<li><a href="./spec/c2.html">InsightUBC v2 specification</a>.</li>
<li><a href="https://github.com/yinhao-f/cpsc-310-2025w2/blob/main/test/dataset/campus.zip"><span>campus.zip</span></a>: building and room information for UBC.</li>
</ul>
<h3>New Requirements</h3>
<p>Building on your Checkpoint 1 implementation, you will:</p>
<ol>
<li><strong>Facilities Data Support</strong>: Process and store UBC building and room data from HTML archives</li>
<li><strong>Advanced Query Features</strong>: Implement aggregation (GROUP with aggregate functions) and multi-key sorting</li>
<li><strong>New REST Endpoints</strong>: Provide direct access to buildings and rooms resources</li>
<li><strong>Data Relationships</strong>: Manage parent-child relationships between buildings and rooms while maintaining consistency with your existing course/section structure</li>
</ol>
<h3>Changes from Checkpoint 1</h3>
<p>Your Checkpoint 1 codebase forms the foundation, but the following enhancements are required:</p>
<ol>
<li>
<p><strong>Facilities Processing Pipeline</strong>: The dataset upload endpoint now accepts two kinds of data:</p>
<ul>
<li><code>course_offerings</code> (existing): Course and section data from JSON files</li>
<li><code>facilities</code> (new): Building and room data parsed from HTML files in a zip archive</li>
</ul>
<p>Your upload processing must route to the appropriate pipeline based on the <code>kind</code> field.</p>
</li>
<li>
<p><strong>API Versioning</strong>: The API now uses versioning to manage certain endpoints:</p>
<ul>
<li>Deprecated v1 endpoints: <code>/api/v1/datasets</code>, <code>/api/v1/datasets/{id}</code>, and <code>/api/v1/search</code> have v2 equivalents but v1 endpoints must be maintained for backwards compatibility.</li>
<li>Current v1 endpoints: <code>/api/v1/courses</code> and <code>/api/v1/courses/{course}/sections</code> remain the standard (no v2 versions)</li>
<li>New v2 endpoints: <code>/api/v2/datasets</code>, <code>/api/v2/search</code>, and <code>/api/v2/buildings/...</code> provide enhanced functionality</li>
</ul>
</li>
<li>
<p><strong>Search Enhancement</strong>: The <code>/api/v2/search</code> endpoint now supports:</p>
<ul>
<li><strong>Facilities as a queryable kind</strong>: Alongside <code>course_offerings</code>, you can now query rooms data</li>
<li><strong>Aggregation/Grouping</strong>: GROUP functionality with aggregate functions (MAX, MIN, AVG, SUM, COUNT)</li>
<li><strong>Multi-key Sorting</strong>: ORDER multiple fields with direction specifiers (UP/DOWN)</li>
<li><strong>Field-specific validation</strong>: Different fields are available depending on the <code>kind</code> being queried</li>
</ul>
</li>
<li>
<p><strong>New Resources</strong>: Buildings and rooms are now first-class REST resources with full CRUD operations:</p>
<ul>
<li><code>/api/v2/buildings</code>: List, create buildings</li>
<li><code>/api/v2/buildings/{building}</code>: Get, update, delete individual buildings</li>
<li><code>/api/v2/buildings/{building}/rooms</code>: List, create rooms within a building</li>
<li><code>/api/v2/buildings/{building}/rooms/{room}</code>: Get, update, delete individual rooms</li>
</ul>
</li>
</ol>

<h4>Part C: Advanced Search Functionality</h4>
<p>Extend your <code>/api/v2/search</code> endpoint to support new query capabilities.</p>
<p><strong>1. Facilities as a Queryable Kind:</strong></p>
<p>Update your search handler to accept <code>kind: "facilities"</code> alongside <code>course_offerings</code>:</p>
<ul>
<li>Different fields are available for each kind:
<ul>
<li><strong>course_offerings</strong>: avg, pass, fail, audit, year (mfields); title, dept, code, instructor (sfields)</li>
<li><strong>facilities</strong>: lat, lon, seats (mfields); name, building, address, number, type, furniture, href (sfields)</li>
</ul>
</li>
</ul>
<p><strong>2. Aggregation/Grouping</strong> (see API specification TRANSFORMATIONS section for full details):</p>
<p>Implement GROUP with aggregate functions in the TRANSFORMATIONS section:</p>
<pre><code class="language-json"><br>{<br>&nbsp; "kind": "facilities",<br>&nbsp; "query": {<br>&nbsp; &nbsp; "WHERE": {},<br>&nbsp; &nbsp; "OPTIONS": {<br>&nbsp; &nbsp; &nbsp; "COLUMNS": ["title", "overallAvg"]<br>&nbsp; &nbsp; },<br>&nbsp; &nbsp; "TRANSFORMATIONS": {<br>&nbsp; &nbsp; &nbsp; "GROUP": ["title"],<br>&nbsp; &nbsp; &nbsp; "APPLY": [{<br>&nbsp; &nbsp; &nbsp; &nbsp; "overallAvg": {<br>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; "AVG": "avg"<br>&nbsp; &nbsp; &nbsp; &nbsp; }<br>&nbsp; &nbsp; &nbsp; }]<br>&nbsp; &nbsp; }<br>&nbsp; }<br>}<br> </code></pre>
<p><strong>Aggregate Functions:</strong></p>
<ul>
<li><code>MAX</code>, <code>MIN</code>: Return the maximum/minimum numeric value</li>
<li><code>AVG</code>: Return numeric result rounded to 2 decimal places using <code>Decimal.js</code> package (already in your dependencies)
<ul>
<li><strong>Important Implementation Details</strong>: You must follow these exact steps:
<ol>
<li>Convert each value to Decimal: <code>new Decimal(num)</code></li>
<li>Add using Decimal's <code>add()</code> method: <code>let total = new Decimal(0); total = total.add(new Decimal(value))</code></li>
<li>Divide by count (NOT as Decimal): <code>let avg = total.toNumber() / numRows</code></li>
<li>Round with <code>toFixed(2)</code> and cast back: <code>Number(avg.toFixed(2))</code></li>
</ol>
</li>
<li>This exact approach ensures consistent floating-point arithmetic across all environments</li>
</ul>
</li>
<li><code>SUM</code>: Return numeric result rounded to 2 decimal places using <code>toFixed(2)</code>, then cast to number</li>
<li><code>COUNT</code>: Return the (whole) number of unique occurrences of a field in the group</li>
</ul>
<p><strong>Important Constraints:</strong></p>
<ul>
<li><span style="text-decoration: line-through;">All GROUP keys must appear in COLUMNS</span> (implementations where this is enforced are accepted, but dropping constraint as it is not included in the spec)</li>
<li>All COLUMNS not in GROUP must use APPLY functions</li>
<li>APPLY keys must be unique across all APPLY operations</li>
<li><code>MAX</code>, <code>MIN</code>, <code>AVG</code>, and <code>SUM</code> must use mfields; <code>COUNT</code> may use any field</li>
</ul>
<p><strong>3. Multi-Key Sorting</strong> (see API specification ORDER section for full details):</p>
<p>Enhance ORDER to support sorting by multiple keys with direction:</p>
<pre><code class="language-json">{
  "OPTIONS": {
    "COLUMNS": ["seats", "address", "type"],
    "ORDER": {
      "dir": "DOWN",
      "keys": ["seats", "address", "type"]
    }
  }
}
</code></pre>
<ul>
<li>Each ORDER key must exist in COLUMNS</li>
<li>Direction <code>UP</code> = ascending, <code>DOWN</code> = descending (applies to all keys)</li>
<li>Keys are applied in order: sort by first key, then second key for ties, etc.</li>
<li>If ORDER is not specified, order is unspecified (existing behavior)</li>
<li><strong>Important</strong>: Use the <code>&lt;</code> operator for comparisons, NOT <code>localeCompare()</code>. The <code>&lt;</code> operator is significantly faster and more consistent across environments. Using <code>localeCompare()</code> can cause performance issues and test failures.</li>
</ul>
<p><strong>4. Validation Errors:</strong></p>
<p>The specification includes validation error examples for aggregation and sorting. Ensure your implementation validates all of the cases listed.</p>
<p><strong>Backwards Compatibility:</strong></p>
<p>Queries without TRANSFORMATIONS or with single-key ORDER should continue to work as before. Your enhancements should not break existing Checkpoint 1 query tests.</p>
<h5>Query Example 1: Aggregation - Find max seats per building</h5>
<pre><code class="language-json">{
  "kind": "facilities",
  "query": {
    "WHERE": {},
    "OPTIONS": {
      "COLUMNS": ["name", "maxSeats"],
      "ORDER": {"dir": "DOWN", "keys": ["maxSeats"]}
    },
    "TRANSFORMATIONS": {
      "GROUP": ["name"],
      "APPLY": [{"maxSeats": {"MAX": "seats"}}]
    }
  }
}
</code></pre>
<p>Expected behavior: Groups all rooms by building name, finds the maximum seats in each building, returns results sorted by max seats descending.</p>
<h5>Query Example 2: Multi-key sorting with WHERE filter</h5>
<pre><code class="language-json">{
  "kind": "course_offerings",
  "query": {
    "WHERE": {"GT": {"avg": 95}},
    "OPTIONS": {
      "COLUMNS": ["year", "dept", "avg"],
      "ORDER": {
        "dir": "UP",
        "keys": ["year", "avg"]
      }
    }
  }
}
</code></pre>
<p>Expected behavior: Filters course offerings with average &gt; 80, returns year/dept/avg, sorts by year ascending, then by average ascending within each year.</p>
