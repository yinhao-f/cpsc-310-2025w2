# C0P1 Building a Robust Test Suite with TDD

<p><em>Please read the project overview before starting. Checkpoint 0 is an individual checkpoint. This phase of the checkpoint is worth 30% the overall checkpoint grade.</em></p>
<div>
<p><strong>Test-Driven Development (TDD)</strong> is a software development practice where tests are written <em>before</em> the implementation. This approach encourages developers to actively engage with the specification early, helping uncover misunderstandings before costly changes are needed. Writing focused, lightweight tests provides a quick and inexpensive way to validate assumptions about the system.</p>
<p>The typical TDD cycle follows this pattern:</p>
<ol>
<li>
<p><strong>Write Tests First</strong><br>Create a set of independent tests that assert every behavior described in the specification. If you discover behaviors that are unclear or unspecified, clarify them with stakeholders.</p>
</li>
<li>
<p><strong>Run the Tests</strong><br>Initially, all tests should fail because the implementation does not yet exist. If any tests pass by default (due to missing assertions), fix them immediately.</p>
</li>
<li>
<p><strong>Implement the Code</strong><br>Write the implementation until all tests pass successfully.</p>
</li>
<li><strong>Refactor<br></strong><strong></strong>Improve the structure of your code, ensuring that all tests pass before and after making changes.</li>
</ol>
</div>

<div>
<h3><strong>Grading Criteria</strong></h3>
<p>Your grade depends on how effectively your test suite detects deviations from the specification.</p>
<p>We will run your tests against multiple implementations:</p>
<ul>
<li>The first implementation will be valid.</li>
<li>Each subsequent implementation will contain one incorrect behaviour (known as a mutant).</li>
</ul>
<p>At least one of your tests that passed on the valid implementation should fail on the invalid one. This demonstrates that your suite is sensitive to deviations from specification.</p>
<p>The invalid implementations will vary from obvious violations to subtle edge cases, requiring careful attention to detail. Focus on building a <strong>robust, specification-driven test suite</strong>, not on targeting the specific mutations used by AutoTest. This will pay off in Phase 2.</p>
