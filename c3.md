<h2>Background</h2>
<p>By the end of Checkpoints 1 and 2, your team has built a substantial REST API with asynchronous dataset processing, layered resource hierarchies, search, aggregation, and persistence. Most teams will also have accumulated some technical debt along the way. That is normal.</p>
<p>In earlier checkpoints, the primary goal was to make the system function correctly and match the specification. In this checkpoint, the goal changes: you will improve the <strong>design</strong> of your implementation without changing its externally observable behaviour. This happens all the time in practice, where many refactoring insights only become meaningful after feeling real friction in the codebase — duplicated logic, tangled responsibilities, difficult tests, and risky changes. C3 is where you convert that lived experience into cleaner structure and stronger engineering judgment.</p>
<p>There are <strong>no new product features</strong> in C3. Instead, you will refactor your code so that it is easier to test, reason about, review, and extend.</p>
<p>Read this assignment completely before making changes. You will be required to submit a graded pull request, so you'll want to plan your work systematically from the start.</p>

<h2>Task 1: Refactor Your Design (40%)</h2>
<p>The REST worksheet introduced one common way to structure an Express application using routes, controllers, middleware, and models. In C3, you will adapt those ideas to your own project.</p>
<p>You are <strong>not</strong> required to copy a single exact folder structure. However, your refactored design must satisfy the architectural constraints below.</p>
<h3>Architectural Constraints</h3>
<p>The constraints below describe the target design. You are not expected to fully restructure every module in your codebase. Instead, your team will declare one feature area as your <strong>graded area</strong> — a complete vertical slice of the application (routes → controller → service → repository) — and apply the constraints consistently throughout it. The graded area is what TAs will evaluate for Task 1.</p>
<p>You may refactor beyond your declared area, and many teams will. But the graded area is the minimum unit of assessment. A team that cleanly separates one feature area end-to-end and applies these principles throughout it will score well. A team that half-applies all nine constraints across every file will not.</p>
<p>Declare your graded area in your design document (see below). Choose one of:</p>
<ul>
    <li><code>v2/search</code> — query validation, execution, and transformation</li>
    <li><code>v2/datasets</code> — upload, processing, and persistence</li>
    <li><code>v2/buildings</code> and <code>v2/rooms</code> — buildings and rooms resource endpoints</li>
    <li><code>v1/courses</code> and <code>v1/sections</code> — courses and sections resource endpoints</li>
</ul>
<p>Your updated design must satisfy all of the following constraints within your declared graded area:</p>
<ol>
    <li>
        <p><strong>Routes are thin.</strong></p>
        <ul>
            <li>Route files define URL paths, HTTP methods, and middleware composition.</li>
            <li>Route handlers should not contain loops, complex conditionals, parsing pipelines, or business rules.</li>
        </ul>
    </li>
    <li>
        <p><strong>Controllers handle HTTP concerns only.</strong></p>
        <ul>
            <li>Controllers translate between Express and the rest of your application.</li>
            <li>They may read <code>req</code>, call a service, choose a status code, write to <code>res</code>, and shape an HTTP response.</li>
            <li>They should not implement core domain logic.</li>
        </ul>
    </li>
    <li>
        <p><strong>Services contain business logic.</strong></p>
        <ul>
            <li>Validation that depends on domain rules, upload processing, search semantics, transformation logic, and coordination across repositories belong here.</li>
            <li>Service functions should be self-contained where practical: given the same inputs, they should produce the same outputs, without depending on global state or Express objects.</li>
        </ul>
    </li>
    <li>
        <p><strong>Repositories are the only layer that touches raw persistence.</strong></p>
        <ul>
            <li>Reading/writing JSON files, disk directories, caches, or in-memory persistence adapters must be isolated here.</li>
            <li>Controllers and services should not directly call <code>fs</code>, path-based storage helpers, or raw data files.</li>
        </ul>
    </li>
    <li>
        <p><strong>Middleware handles cross-cutting concerns.</strong></p>
        <ul>
            <li>Examples: request validation, error handling, request logging, upload parsing, shared parameter parsing.</li>
            <li>Repeated HTTP plumbing should move into middleware rather than being copied across endpoints.</li>
        </ul>
    </li>
    <li>
        <p><strong>Domain models are explicit.</strong></p>
        <ul>
            <li>The core objects your application works with (e.g. courses, sections, buildings, rooms, queries) should be represented deliberately using interfaces, types, or small domain modules.</li>
            <li>Domain models should not depend on Express.</li>
            <li>Domain models should not be responsible for raw file I/O.</li>
        </ul>
    </li>
    <li>
        <p><strong>Dependencies are one-directional.</strong></p>
        <ul>
            <li>Avoid circular imports.</li>
            <li>A typical dependency direction is:
                <ul>
                    <li>routes → controllers → services → repositories → fs/storage</li>
                </ul>
            </li>
        </ul>
    </li>
    <li>
        <p><strong>Business logic is not duplicated.</strong></p>
        <ul>
            <li>If the same rule appears in multiple endpoints, extract it.</li>
            <li>“Shared utility” modules are acceptable only when they are genuinely shared abstractions rather than a dumping ground.</li>
        </ul>
    </li>
    <li>
        <p><strong>Behaviour is preserved by tests.</strong></p>
        <ul>
            <li>Your refactor should leave existing behaviour intact.</li>
            <li>If you change structure aggressively, your test suite should give you confidence that the system still works.</li>
        </ul>
    </li>
</ol>
<h3>Required Design Decisions</h3>
<p>As part of the refactor, your team must make and justify the following design decisions in a short design document.</p>
<p><strong>For each decision below, choose one approach, explain why you chose it, and describe one tradeoff.</strong></p>
<h4>1. How will services access repositories?</h4>
<p>Choose one:</p>
<ul>
    <li><strong>Direct imports</strong>: service modules import repository modules directly</li>
    <li><strong>Dependency injection</strong>: repositories are passed into services/functions</li>
    <li><strong>Factory/composition root</strong>: wiring happens in one setup module that assembles controllers/services/repositories</li>
</ul>
<h4>2. How will you structure validation?</h4>
<p>Choose one:</p>
<ul>
    <li><strong>Per-route schema modules</strong>: each endpoint or resource owns its validation rules</li>
    <li><strong>Centralized schema registry</strong>: request schemas live in one shared validation area</li>
    <li><strong>Middleware-first validators</strong>: validation is composed as reusable middleware functions</li>
</ul>
<h4>3. How will you represent domain errors?</h4>
<p>Choose one:</p>
<ul>
    <li><strong>Custom <code>Error</code> subclasses</strong>: throw typed errors (e.g. <code>NotFoundError</code>, <code>ValidationError</code>, <code>TooManyResultsError</code>); callers use <code>try/catch</code></li>
    <li><strong>Result-style return values</strong>: return <code>Ok</code> / <code>Err</code> values instead of throwing exceptions for domain errors</li>
</ul>
<h4>4. How will you organize your modules?</h4>
<p>Choose one:</p>
<ul>
    <li><strong>Per-resource</strong>: courses, sections, buildings, rooms, datasets, search</li>
    <li><strong>Per-layer</strong>: all controllers together, all services together, all repositories together</li>
    <li><strong>Hybrid feature-first</strong>: each feature owns its controllers/services/repositories, plus a small shared core</li>
</ul>
<p>Your design document should make it easy for a reviewer to understand your architecture without having to infer it from the code alone.</p>

<h5>4. Repository isolated from storage details</h5>
<pre><code class="language-ts">export async function listByCourse(courseId: string, limit: number, offset: number) {
  const allSections = await fileStore.readCollection("sections");
  return allSections
    .filter((section) =&gt; section.courseId === courseId)
    .sort((a, b) =&gt; (a.id &lt; b.id ? -1 : a.id &gt; b.id ? 1 : 0))
    .slice(offset, offset + limit);
}

<h5>6. Error-handling middleware</h5>
<pre><code class="language-ts">export function handleErrors(err: unknown, _req: Request, res: Response, _next: NextFunction) {
  if (err instanceof NotFoundError) {
    return res.status(404).json({ error: "Not found", message: err.message });
  }

  if (err instanceof ValidationError) {
    return res.status(422).json({ error: "Validation failed", message: err.message });
  }

  res.status(500).json({ error: "Internal server error" });
}
</code></pre>

<p><strong>Focus on the diff.</strong> Open the PR's "Files changed" tab and review only what changed. You are not being asked to audit the entire codebase — you are evaluating whether this specific set of changes is a good architectural move. If a file appears in the diff but most of its lines are unchanged, focus your attention on the lines that are new or removed.</p>
<p>At minimum, your review should address:</p>
<ul>
    <li>
        <p><strong>Separation of concerns.</strong> Does each module do exactly one job? For example: does the route handler contain business logic that belongs in a service? Does a controller call <code>fs</code> directly instead of going through a repository? Does a service read from <code>req</code> or write to <code>res</code>?</p>
    </li>
    <li>
        <p><strong>Dependency direction.</strong> Dependencies should flow in one direction: routes depend on controllers, controllers depend on services, services depend on repositories. Nothing should point backward. Check whether any inner layer imports from an outer layer — for example, a service that imports a controller, or a repository that imports a service.</p>
    </li>
    <li>
        <p><strong>Duplication.</strong> Is the same validation rule, query logic, or error-handling pattern repeated across multiple endpoints or modules? Where should it live instead?</p>
    </li>
    <li>
        <p><strong>Testability of the changed layer.</strong> Can the service or repository now be tested without spinning up an Express app? Can a controller be tested by passing mock <code>req</code>/<code>res</code> objects? Is the changed code easier or harder to unit-test than before?</p>
    </li>
    <li>
        <p><strong>Behavioural safety.</strong> Does the refactor appear to preserve existing behaviour? Look for specific risks: a renamed function that silently changes its return shape, an error that previously produced a 404 but might now produce a 500, a missing <code>await</code> on an async call, or middleware that previously ran on a route but is no longer registered.</p>
    </li>
</ul>
<p>A strong review points out tradeoffs, not just mistakes.</p>

<h2>Task 3: Regression Tests (30%)</h2>
<p>We will re-run the C1 and C2 autograder tests against your refactored implementation.</p>
<p>The purpose of this component is to ensure that your refactor did not break existing functionality.</p>
<p>Task 3 does not require new functionality. It re-runs the existing C1/C2 tests to confirm that your refactor preserved system behaviour across the project.</p>
<p>Teams may improve previously failing behaviour during C3. If those fixes align with the published specification, the improved test results will not count as regressions. You will not be penalized for functionality that was not already working at the end of the corresponding checkpoint.</p>
<p>You should expect us to evaluate whether your project still supports:</p>
<ul>
    <li>the C1 dataset upload, course/section endpoints, and search behaviour</li>
    <li>the C2 facilities upload, buildings/rooms endpoints, and v2 search behaviour</li>
    <li>existing error cases and status code behaviour</li>
</ul>
<h3>Practical Guidance</h3>
<ul>
    <li>Run your existing test suite frequently during refactoring</li>
    <li>Prefer small, staged refactors over one giant rewrite</li>
    <li>If you move code across files, keep the same tests running throughout</li>
    <li>Add focused tests where your old design had weak coverage</li>
</ul>
<p>Remember: a clean architecture that breaks the spec is still an incorrect submission.</p>

<h2>Suggested Refactor Strategy</h2>
<p>You are not required to follow this exact order, but many teams will find the following sequence effective:</p>
<ol>
    <li>
        <p>Identify the worst pain points in your current design</p>
        <ul>
            <li>giant route handlers</li>
            <li>repeated validation</li>
            <li>duplicated search logic</li>
            <li>raw file access scattered across the codebase</li>
        </ul>
    </li>
    <li>
        <p>Choose one architectural seam first</p>
        <ul>
            <li>extract repositories</li>
            <li>add centralized error handling</li>
            <li>split controllers from services</li>
        </ul>
    </li>
    <li>
        <p>Stabilize with tests</p>
        <ul>
            <li>make sure behaviour is still preserved</li>
        </ul>
    </li>
    <li>
        <p>Repeat with a second or third targeted refactor</p>
    </li>
    <li>
        <p>Select one coherent subset for your authored PR</p>
    </li>
</ol>
<p>This checkpoint rewards systematic improvement more than heroic rewrites.</p>
