# Checkpoint 1

<h2 id="background" class="code-line" data-line="2">Background</h2>
<p class="code-line" data-line="4">Congratulations! The Decision Support team in the UBC Registrar’s Office happened to see your UniFlow demo (Checkpoint 0), and after a few follow-up chats, they've invited your team to build a full production system called<span>&nbsp;</span><em><strong>InsightUBC</strong></em>. This system will enable UBC administrators and researchers to analyze historical course offering data collected by the PAIR team.</p>
<h3 id="new-requirements" class="code-line" data-line="6">New Requirements</h3>
<p class="code-line" data-line="8">The client has specified the following core functionality:</p>
<ol class="code-line" data-line="10">
<li class="code-line" data-line="10"><strong>Bulk Dataset Upload</strong>: Accept and process historical course offerings data provided as zip files. An example dataset containing <strong>64,612</strong> valid&nbsp;sections is available<span>&nbsp;</span><a href="../test/dataset/pair.zip">here</a>.</li>
<li class="code-line" data-line="11"><strong>Advanced Querying</strong>: Support complex queries over course offerings data to enable analysis and reporting.</li>
</ol>
<h3 id="changes-from-checkpoint-0" class="code-line" data-line="13">Changes from Checkpoint 0</h3>
<p class="code-line" data-line="15">While much of your UniFlow code can be reused, winning this contract has driven several important architectural changes:</p>
<ol class="code-line" data-line="17">
<li class="code-line code-active-line" data-line="17">
<p class="code-line" data-line="17"><strong>Single Organization Focus</strong>: The application is now built exclusively for UBC, eliminating the need to support multiple organizations. The<span>&nbsp;</span><code>/orgs</code><span>&nbsp;</span>route and organization-level abstractions have been removed from the API.</p>
</li>
<li class="code-line" data-line="19">
<p class="code-line" data-line="19"><strong>Scalability Improvements</strong>: With datasets containing 64,000+ sections, your REST API must now handle large-scale data efficiently:</p>
<ul class="code-line" data-line="20">
<li class="code-line" data-line="20"><strong>Pagination</strong>: GET endpoints for courses and sections must support offset-based pagination with configurable limits</li>
<li class="code-line" data-line="21"><strong>Link Optimization</strong>: Course responses now return a single link to the sections endpoint rather than embedding arrays of section URLs</li>
<li class="code-line" data-line="22"><strong>Efficient Deletes</strong>: DELETE operations return only metadata (e.g., count of deleted items) rather than full resource representations</li>
</ul>
</li>
</ol>
<p class="code-line" data-line="24">&nbsp;The updated API specification is available <a href="./spec/c1.html">here</a>.</p>

<h4 id="query-language" class="code-line" data-line="121">Query Language</h4>

<ul class="code-line" data-line="125">

<p class="code-line" data-line="130">See the<span>&nbsp;</span>specification<span>&nbsp;</span>for complete syntax details and multiple working examples in the dropdown menu.</p>

<li class="code-line" data-line="161">
<p class="code-line" data-line="161"><strong>Pagination</strong>: Implement pagination for list endpoints (GET /courses, GET /sections).</p>
<p class="code-line" data-line="163"><strong>What is pagination?</strong><span>&nbsp;</span>When you have thousands of results, sending them all at once is inefficient. Pagination breaks results into "pages" using<span>&nbsp;</span><code>limit</code><span>&nbsp;</span>(items per page) and<span>&nbsp;</span><code>offset</code><span>&nbsp;</span>(starting position). For example, with 1000 items, a<span>&nbsp;</span><code>limit</code><span>&nbsp;</span>of 100 and<span>&nbsp;</span><code>offset</code><span>&nbsp;</span>of 200 returns items 200-299 (the third page).</p>
<p class="code-line" data-line="165"><strong>Query parameters</strong><span>&nbsp;</span>are values passed in the URL after a<span>&nbsp;</span><code>?</code><span>&nbsp;</span>(e.g.,<span>&nbsp;</span><code>/courses?limit=100&amp;offset=200</code>). They're accessed in Express via<span>&nbsp;</span><code>req.query</code>. Multiple parameters are separated by<span>&nbsp;</span><code>&amp;</code>.</p>
<p class="code-line" data-line="167"><strong>Stable ordering</strong>: For pagination to be reliable across multiple requests, list endpoints must return results in a consistent order. Use ascending<span>&nbsp;</span><code>id</code><span>&nbsp;</span>ordering for courses and sections.</p>
<p class="code-line" data-line="169"><strong>Simple implementation</strong><span>&nbsp;</span>using<span>&nbsp;</span><code>array.slice()</code>:</p>

<h4 id="testing-strategy" class="code-line" data-line="214">Testing Strategy</h4>
<p class="code-line" data-line="216"><strong>Your test suite is the single most important tool for reducing development time and effort.</strong><span>&nbsp;</span>A comprehensive test suite catches bugs immediately, allows you to refactor confidently, and validates your implementation before submitting to AutoTest. Writing tests as you implement features is far more efficient than debugging failing AutoTest runs.</p>
<p class="code-line" data-line="218"><strong>Recommended Approach</strong>:</p>
<ol class="code-line" data-line="220">
<li class="code-line" data-line="220">
<p class="code-line" data-line="220"><strong>Start with Black-Box Tests</strong><span>&nbsp;</span>(just like <a href="./c0p1.md">C0P1</a>): Write tests based on the specification without looking at your implementation. Use<span>&nbsp;</span><strong>equivalence class partitioning</strong><span>&nbsp;</span>to systematically identify test cases:</p>
<p class="code-line" data-line="222"><strong>Input Partitioning (Requests)</strong>: Group similar inputs that should be handled the same way:</p>
<ul class="code-line" data-line="223">
<li class="code-line" data-line="223">Valid inputs (well-formed requests, valid IDs, proper query syntax)</li>
<li class="code-line" data-line="224">Invalid inputs (malformed data, missing required fields, invalid types)</li>
</ul>
<p class="code-line" data-line="226"><strong>Output Partitioning (Responses)</strong>: Test each response class your API can produce considering both the HTTP status codes and content:</p>
<ul class="code-line" data-line="227">
<li class="code-line" data-line="227">Success vs error responses</li>
<li class="code-line" data-line="228">Collections with exactly 1 item vs. many items (tests iteration logic, system state)</li>
<li class="code-line" data-line="229">Empty vs. non-empty result sets (<code>{"items": [], "total": 0}</code><span>&nbsp;</span>vs.<span>&nbsp;</span><code>{"items": [...], "total": 5}</code>)</li>
</ul>
<p class="code-line" data-line="231">Combine equivalence partitioning with<span>&nbsp;</span><strong>boundary value analysis</strong><span>&nbsp;</span>to identify edge cases:</p>
<ul class="code-line" data-line="232">
<li class="code-line" data-line="232">Input boundaries: empty strings, very large datasets, null values, maximum/minimum numeric values</li>
<li class="code-line" data-line="233">Pagination boundaries (offset=0, offset=total, limit=0, limit=max)</li>
<li class="code-line" data-line="234">Empty datasets (no courses, no sections)</li>
<li class="code-line" data-line="235">Complex queries (deeply nested filters, AND/OR combinations)</li>
<li class="code-line" data-line="236">Maximum result size</li>
</ul>
<p class="code-line" data-line="238">Black-box tests ensure your API meets the specification and provide a safety net as you refactor.</p>
</li>
<li class="code-line" data-line="240">
<p class="code-line" data-line="240"><strong>Strengthen with Glass-Box Tests</strong>: Once you have an initial implementation, run<span>&nbsp;</span><code>yarn cover</code><span>&nbsp;</span>to see your code coverage:</p>
<pre><code class="code-line language-bash" data-line="241">yarn cover
</code></pre>
<p class="code-line" data-line="245">This generates:</p>
<ul class="code-line" data-line="246">
<li class="code-line" data-line="246">Console output with coverage percentages</li>
<li class="code-line" data-line="247">An HTML report in<span>&nbsp;</span><code>coverage/</code><span>&nbsp;</span>(open<span>&nbsp;</span><code>coverage/index.html</code><span>&nbsp;</span>in your browser)</li>
</ul>
<p class="code-line" data-line="249">Use coverage reports to identify untested code paths. Add targeted tests for:</p>
<ul class="code-line" data-line="250">
<li class="code-line" data-line="250">Branches you haven't exercised</li>
<li class="code-line" data-line="251">Error handling paths</li>
<li class="code-line" data-line="252">Edge cases in your implementation</li>
</ul>
<p class="code-line" data-line="254">Aim for &gt;80% coverage, but remember: coverage is a metric, not a goal. High coverage with poor assertions is useless.</p>
</li>
</ol>
<p class="code-line" data-line="256"><strong>Unit Tests vs Integration Tests</strong></p>
<p class="code-line" data-line="258">A common testing anti-pattern is writing only integration tests (tests that call your top-level API methods like<span>&nbsp;</span><code>POST /datasets</code><span>&nbsp;</span>or<span>&nbsp;</span><code>POST /search</code>). While these are important, they're slow and make fault isolation difficult.</p>
<p class="code-line" data-line="260"><strong>Better approach</strong>: Write unit tests for individual methods and classes:</p>
<ul class="code-line" data-line="261">
<li class="code-line" data-line="261">Test your data parsing logic independently</li>
<li class="code-line" data-line="262">Test query validation separately from query execution</li>
<li class="code-line" data-line="263">Test filtering logic without loading full datasets</li>
<li class="code-line" data-line="264">Test pagination logic with small arrays</li>
</ul>
<p class="code-line" data-line="266">This makes your test suite:</p>
<ul class="code-line" data-line="267">
<li class="code-line" data-line="267"><strong>Faster</strong>: Unit tests run in milliseconds</li>
<li class="code-line" data-line="268"><strong>More reliable</strong>: Better at detecting specific faults</li>
<li class="code-line" data-line="269"><strong>Easier to debug</strong>: Failures point directly to the broken component</li>
</ul>
<p class="code-line" data-line="271">💡<span>&nbsp;</span><strong>Testing Tip</strong>: Create small zip files with just a few courses/sections for testing the dataset upload endpoint. For search endpoint testing, load minimal data to validate your query logic (e.g., 2-3 courses with 5-10 sections each). This makes debugging much faster than using the full 65,000-section dataset.</p>

<h2 id="tips-for-success" class="code-line" data-line="431">Tips for Success</h2>
<ul class="code-line" data-line="433">
<li class="code-line" data-line="433"><strong>Start Early</strong>: This checkpoint requires significantly more design and implementation work than C0. Start immediately to allow time for questions and iteration</li>
<li class="code-line" data-line="434"><strong>Implement Backend First</strong>: Get your endpoints working before starting your frontend—the frontend depends on a functioning API</li>
<li class="code-line" data-line="435"><strong>Create Test Datasets</strong>: Build small zip files with 5-10 sections for rapid testing—don't use the full dataset until you're confident your code works</li>
<li class="code-line" data-line="436"><strong>Test Incrementally</strong>: Don't wait until everything is built to start testing. Test each component as you build it</li>
<li class="code-line" data-line="437"><strong>Use the Spec Examples</strong>: The specification includes multiple query examples—test your implementation against all of them</li>
<li class="code-line" data-line="438"><strong>Handle Edge Cases</strong>: Empty datasets, invalid queries, boundary conditions for pagination</li>
<li class="code-line" data-line="439"><strong>Follow Git Workflow</strong>: Make small, focused commits and open PRs early for feedback</li>
<li class="code-line" data-line="440"><strong>Review Thoroughly</strong>: Take code review seriously—catch bugs before they reach<span>&nbsp;</span><code>main</code></li>
<li class="code-line" data-line="441"><strong>Merge Regularly</strong>: Don't let feature branches diverge too far from<span>&nbsp;</span><code>main</code></li>
<li class="code-line" data-line="442"><strong>Use AutoTest Feedback Wisely</strong>: You have 3 requests per day—use them strategically after making significant changes</li>
<li class="code-line" data-line="443"><strong>Communicate</strong>: Regular check-ins with your partner prevent integration headaches</li>
<li class="code-line" data-line="444"><strong>Ask Questions</strong>: Use Piazza or office hours if you're unclear about requirements</li>
</ul>
